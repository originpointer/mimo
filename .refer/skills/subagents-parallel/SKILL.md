---

## 指令

- 先识别任务是否适合并行：
  - 需要多条独立探索（多文件/多模块/多方案对比）
  - 或需要多种视角（实现/测试/文档/风险评估）
- 设计 2-4 个子任务，要求：
  - 每个子任务目标单一、边界清晰
  - 明确输入（上下文/文件/接口）与输出（结论/patch/列表）
  - 避免重复劳动（例如每个子代理都全仓库搜索）
- 给出“主代理协调策略”：
  - 并行执行顺序
  - 结果合并方式（冲突处理、去重、最终决策）

## 输出格式

- 是否适合并行：是/否 + 原因
- 子任务拆分：
  - Subagent A：目标 / 输入 / 输出 / 验收点
  - Subagent B：...
  - Subagent C：...
- 主代理合并：
  - 合并规则：...
  - 冲突处理：...
  - 最终交付：...

## 常见坑

- 子任务边界不清，导致相互依赖、无法并行。
- 子代理输出格式不统一，主代理难以合并。
- 只追求并行数量，忽略“可验证的验收点”。
